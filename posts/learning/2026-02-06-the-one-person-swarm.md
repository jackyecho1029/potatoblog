很多做独立站或者折腾个人项目的朋友，最近跟我聊起 AI 都有一种疲惫感：

“每天跟 AI 聊天聊得口干舌燥，代码改了这头忘了那头，感觉 AI 只有脑子没有手，最后还是得我一个一个文件去粘。”

**听起来熟悉吗？这就是典型的“把顶级 Agent 当成普通的聊天机器人”在用。**

最近我研究了 AI 专家 Allie Miller 提到的一种 **“单人蜂群（One-Person Swarm）”** 架构，并实测了一套配置方法。今天不聊大道理，直接上干货，教大家怎么让你的 Claude Code 开启“分身模式”，多任务并行作业。

---

### 🔍 痛点聚焦：为什么你的 AI 效率上不去？

绝大多数人的用法是：开一个窗口，问一个问题，等 AI 回答，然后自己手动测试。
**这个逻辑的硬伤在于：你把宝贵的“人脑”时间，锁定在了一个“单线程”的流程里。**

其实真正的进阶玩法，是利用 **Git Worktree + CLAUDE.md**，实现物理级的任务并行。

---

### 🛠️ 工具实战：三步开启“分身术”

#### 1. 物理分身：Git Worktree (核心黑科技)
如果你在主分支让 AI 改样式，你自己想去改逻辑，往往会发生冲突。
**解决方法：** 用 Git 的“分身术”。
在你的项目目录运行：
```bash
git worktree add ../feature-branch feature-branch
```
**这时候，你的电脑里会出现两个完全隔离的项目文件夹。**
你可以在 A 文件夹启动一个 Claude 处理文案，在 B 文件夹启动另一个 Claude 跑测试。这才是真正的并行，效率直接翻倍。

#### 2. 给每个分身装上“大脑”：CLAUDE.md
这是 Allie Miller 特别强调的一点。你需要在项目根目录放一个 `CLAUDE.md`，像写说明书一样告诉 AI 它在这个文件夹里的角色。

**这是一份可以直接复制的模板：**
```markdown
# Role: Senior Full-Stack Developer
- Project Style: Minimalist & Fast
- Rules: Before coding, MUST run 'npm run check'.
- Goals: Automate repetitive tasks first.
```
有了这个文件，你每次打开 Claude Code，它不需要你重新调教，直接进入状态。

#### 3. 调度指令：从“提问”转向“指挥”
别再说“帮我改下这个 Bug”。试着下达这种**复合指令**：
> “分析 `posts/` 下的风格，同步生成 3 篇稿件，并自动写入 `drafts/` 目录，最后运行测试确保无误。”

---

### ⚠️ 实战避坑：别让“蜂群”变成“乱群”

很多人尝试并行之后发现，AI 还是会乱。我总结了两个血泪教训：

1.  **上下文隔离**：每个 Worktree 至少要有独立的任务目标。别让两个 AI 同时去改同一个公共组件，否则最后代码合并时你会想哭。
2.  **别省那一分钟的 Plan 时间**：在让 AI 开始大干苦干前，先进入 `Plan` 模式（双击 Tab）。让它把要干的事先列出来，你确认没问题再放它去跑。

### 🎯 总结建议

小渔经常说，独立站运营不要追求花里胡哨，要追求**“单位时间的产出”**。

开启这个“蜂群模式”后，你就不再是一个敲代码的苦力，而是一个调度 AI 算力的机器。你负责定规则（CLAUDE.md），负责分发环境（Worktree），AI 负责执行。
